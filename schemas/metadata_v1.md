# NIDABA Metadata Schema (v1.0)
**Project:** NIDABA — The Scribe & Archivist
**Purpose:** Defining the semantic tags for atomizing a high-redundancy, human-centric corpus.

---

## 1. Top-Level Topics
Every atom must be assigned to at least one primary topic:
- **IDENTITY:** Naming, Cultural Roots, Personal Philosophy, Values, "Building Persona."
- **ASSETS:** Hardware (Nodes, Spares), Software (Nisaba, Enki), Infrastructure (Network, Storage).
- **FRAMEWORKS:** Design Patterns, Decision Logic, Game Rubrics, Operational Rules.
- **COMMONPLACE:** Literature, History, Speculative Humanism, Cooking, Life Systems.
- **CREATIVE:** Fiction drafts, Character notes, Narrative logic, Worldbuilding.

## 2. Epistemic Types (How to use this data)
- **FACT:** Verifiable data points (e.g., "Atlas has 32GB RAM"). Audit against reality.
- **RATIONALE:** The "Why" behind a system (e.g., "Why we use JSONL"). Preserve as intent.
- **HEURISTIC:** Decision-making rules or filters (e.g., "If 3 YES, then wishlist").
- **LORE:** Philosophical anchors and creative references (e.g., "Solaris themes").

## 3. Lifecycle States
- **ACTIVE:** The current, authoritative "Source of Truth."
- **LEGACY:** Historical state or previous version (to be archived).
- **DEPRECATED:** Dead-end projects or rejected ideas (for reference only).
- **ITERATIVE:** Conversational transcripts, rubber-ducking, and brainstorms.

## 4. Source Mapping (The "Audit Trail")
Each atom must retain:
- `uuid`: SHA256 hash of content (deterministic for de-duplication).
- `source_path`: Relative path from staging root.
- `line_range`: [start, end] for precise reference.
- `timestamp`: ISO-8601 (extracted from filename or metadata).
- `role`: (User/Assistant) for transcripts.
